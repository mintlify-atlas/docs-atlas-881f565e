## Directory Reference

### bin/

Custom binaries and executable symlinks that should be available system-wide.

<Accordion title="Why use bin/?">
  The `bin/` directory is added to your PATH with high priority, making any executables here available from anywhere in your terminal. This is perfect for:
  
  - Custom scripts you use frequently
  - Symlinks to executables from other tools
  - Wrapper scripts that modify tool behavior
</Accordion>

**Example use cases:**

```bash
# Create a custom script
touch $DOTFILES_PATH/bin/mycommand
chmod +x $DOTFILES_PATH/bin/mycommand

# Now available everywhere:
mycommand
```

**PATH configuration:**

```bash
# From shell/exports.sh
path=(
  "$HOME/bin"              # Personal bin (highest priority)
  "$DOTLY_PATH/bin"       # dotly commands
  "$DOTFILES_PATH/bin"    # Your custom binaries в†ђ This directory
  # ... other paths
)
```

### restoration_scripts/

Executable scripts that run during `dot self install`.

<Note>
  These scripts are executed in alphabetical order during installation, allowing you to set up your environment automatically.
</Note>

**Example scripts:**

```bash

в”њв”Ђв”Ђ 01_install_packages.sh
в”њв”Ђв”Ђ 02_setup_languages.sh
в”њв”Ђв”Ђ 03_clone_repos.sh
в””в”Ђв”Ђ 99_final_setup.sh
```

**Example restoration script:**

```bash
#!/usr/bin/env bash
# restoration_scripts/01_install_packages.sh

source "$DOTLY_PATH/scripts/core/_main.sh"

output::answer "Installing custom packages..."

if platform::is_macos; then
  brew install jq ripgrep fd bat
elif platform::is_linux; then
  sudo apt-get install -y jq ripgrep fd-find bat
fi

output::answer "Packages installed!"
```

**Execution during installation:**

```bash
# From scripts/self/install (lines 43-53)
install_scripts_path="$DOTFILES_PATH/restoration_scripts"
find "$install_scripts_path" -mindepth 1 -maxdepth 1 -type l,f -name '*.sh' |
  sort |
  while read -r install_script; do
    [[ -x $install_script ]] && . "$install_script"
  done
```

### scripts/

Custom executable scripts organized by context.

**Default contexts:**

```bash
scripts/

в”њв”Ђв”Ђ package/                # Package management
```

**Creating custom contexts:**

```bash
# Create a new context
mkdir -p $DOTFILES_PATH/scripts/deploy

# Add a script
cat > $DOTFILES_PATH/scripts/deploy/staging << 'EOF'
#!/usr/bin/env bash
source "$DOTLY_PATH/scripts/core/_main.sh"

##? Deploy to staging environment
##?
##? Usage:
##?    deploy staging
docs::parse "$@"

output::answer "Deploying to staging..."
# Your deployment logic
EOF

chmod +x $DOTFILES_PATH/scripts/deploy/staging
```

Now run with: `dot deploy staging`

<Tip>
  Scripts in `$DOTFILES_PATH/scripts/` override scripts with the same name in `$DOTLY_PATH/scripts/`, allowing you to customize default dotly behavior.
</Tip>

### symlinks/

Configuration files that define symbolic links between your dotfiles and their target locations.

**Available files:**

```bash
symlinks/
в”њв”Ђв”Ђ conf.yaml               # Universal symlinks (all OS)
в”њв”Ђв”Ђ conf.macos.yaml         # macOS-specific symlinks
в””в”Ђв”Ђ conf.linux.yaml         # Linux-specific symlinks
```

**Format:**

```yaml
# Target location: Source in dotfiles
~/.gitconfig: git/.gitconfig
~/.vimrc: editors/vim/.vimrc
~/Library/Application Support/Code/User/settings.json: editors/code/settings.json
```

**Apply symlinks:**

```bash
dot symlinks apply
```

<Accordion title="How symlinks work">
  When you run `dot symlinks apply`, dotly:
  
  1. Reads the appropriate `conf.*.yaml` file for your OS
  2. Creates symbolic links from the target location to your dotfiles
  3. Backs up any existing files before creating symlinks
  4. Creates parent directories if they don't exist
  
  This means changes to files in your dotfiles are immediately reflected in their target locations, and vice versa.
</Accordion>

## Best Practices

<Steps>
  <Step title="Organize by purpose">
    Keep related configurations together. Put editor configs in `editors/`, language configs in `langs/`, etc.
  </Step>
  
  <Step title="Use OS-specific folders when needed">
    If a configuration differs between macOS and Linux, create OS-specific subdirectories or symlink files.
  </Step>
  
  <Step title="Document your custom setup">
    Add README files in `doc/` explaining non-obvious configurations and custom scripts.
  </Step>
  
  <Step title="Commit frequently">
    Commit changes to your dotfiles repository after adding or modifying configurations.
  </Step>
  
  <Step title="Test restoration scripts">
    Periodically test your `restoration_scripts/` in a clean environment to ensure they work correctly.
  </Step>
</Steps>
